# Final Project Proposal Presentation Guidelines

The Final Project Proposal presentation lets you present your proposed system, identify the real problem faced by your chosen client, and explain how your team plans to solve it. Your proposal must show a clear problem, a logical solution, and a practical plan for building the system.

---

## 1. Team Composition

- Each group must have **1–4 members only**.
- All members must participate in the presentation.
- Each member must have a clear responsibility in the project.

---

## 2. Required Parts of the Presentation

Your presentation must answer the following questions clearly.

### 2.1 Client Identification

Identify the organization, office, or individual that will benefit from your system.

**Include:**
- Name of the organization/client
- Type of organization (school, business, government office, etc.)
- Short background about the client
- Current process they use

**Example:**
> "The client is the Registrar Office of XYZ College. They currently manage student records manually using spreadsheets."

### 2.2 Problem Identification

Your proposal must clearly answer these questions.

#### What is the problem?

Describe the specific issue or inefficiency currently experienced by the client.

**Examples:**
- Manual record keeping
- Slow processing of requests
- Lost or inconsistent data
- Lack of monitoring system

#### Why is it a problem?

Explain the impact of the problem.

**Examples:**
- Causes delays in service
- Leads to inaccurate records
- Reduces productivity
- Creates inconvenience for users

> Use real observations or interviews if possible.

### 2.3 Proposed Solution

**What is your solution?**

Present the system you want to develop.

**Include:**
- System name
- Type of system (web system, mobile app, IoT system, etc.)
- Short description of how it solves the problem

**Example:**
> "We propose a Web-Based Appointment Management System that allows students to schedule consultations online."

### 2.4 How Will You Develop the Solution?

Explain how your team will create the system.

**Include:**
- Development tools or technologies
  - Example: React, PHP, Firebase, Python, IoT devices
- Method you will use
  - Example: Agile, Prototype Model, Iterative development
- Steps you will take
  - Example development steps:
    1. Problem analysis
    2. Requirements gathering
    3. System design
    4. System development
    5. Testing
    6. Deployment

### 2.5 Proposed System Features

List the major features of your system.

**Example format:**

**User Features**
- User registration and login
- Data submission
- Dashboard

**Admin Features**
- Manage users
- Generate reports
- Monitor system activity

> Focus on practical and realistic features.

---

## 3. Team Member Roles

Each group member must have a defined role.

| Role | Responsibilities |
|------|------------------|
| Project Leader/Hustler | Manages the project, coordinates the team, communicates with the client |
| System Analyst/Document | Identifies the problem, gathers requirements, documents system needs |
| Developer / Programmer | Develops the system, implements features, writes the code |
| UI/UX Designer / Tester | Designs the interface and performs system testing |

> If the team has fewer members, responsibilities can be combined.

---

## 4. Presentation Structure

Follow this order in your slides:

1. Title of the Proposed System
2. Team Members
3. Client Background
4. Current Process
5. Problem Identification
6. Why It Is a Problem
7. Proposed System Solution
8. System Features
9. Development Plan
10. Technologies to be Used
11. Expected Benefits
12. Conclusion

---

## 5. Presentation Rules

- **Presentation Time:** 5–10 minutes per group
- **All members must speak** during the presentation
- Use clear and professional slides
- Avoid too much text
- Include diagrams if possible

---

## 6. Evaluation Criteria

| Criteria | Description |
|----------|-------------|
| Problem Identification | Clear explanation of the real problem |
| Relevance to Client | Solution directly addresses the client's needs |
| Feasibility of System | The system can realistically be developed |
| System Features | Features are useful and well-defined |
| Presentation Quality | Clear explanation and organized slides |
| Team Participation | All members contribute to the presentation |

---

## Rubric

| Criteria | 4 – Excellent | 3 – Good | 2 – Fair | 1 – Poor | Points |
|----------|---------------|----------|----------|----------|--------|
| **Client Identification** | Client clearly identified with strong background and current process explained | Client identified but background limited | Client mentioned but explanation unclear | Client not clearly identified | /4 |
| **Problem Identification** | Problem clearly defined, specific, and supported with explanation | Problem identified but explanation limited | Problem vague or partially explained | Problem unclear or missing | /4 |
| **Why It Is a Problem** | Strong justification showing impact to the client | Impact explained but lacks depth | Weak explanation of impact | No clear explanation | /4 |
| **Proposed Solution** | Solution clearly addresses the problem and is logical | Solution mostly relevant to the problem | Solution somewhat related | Solution unclear or unrelated | /4 |
| **System Features** | Features are clear, organized, and realistic | Features listed but lack detail | Few features identified | Features unclear or missing | /4 |
| **Development Plan** | Clear explanation of how the system will be developed including tools and process | Development process mentioned but limited | Plan unclear or incomplete | No development plan explained | /4 |
| **Feasibility of Project** | System appears realistic and achievable | Mostly feasible with minor concerns | Feasibility questionable | System unrealistic | /4 |
| **Presentation Organization** | Presentation well structured, clear flow of ideas | Mostly organized | Some organization issues | Disorganized presentation | /4 |
| **Team Participation** | All members actively participate and explain their parts | Most members participate | Only few members present | One person dominates the presentation | /4 |
| **Ability to Answer Questions** | Answers questions clearly and confidently | Answers most questions | Limited answers | Cannot answer questions | /4 |
